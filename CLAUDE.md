# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`edge-inventory` is a FOLIO **edge module**: a Spring Boot 4 / Java 21 service that exposes selected, read-only FOLIO inventory APIs to external systems. External callers authenticate with an edge **API key**; the module resolves it to an institutional user, obtains an Okapi token, and proxies the request to internal FOLIO services. Almost all authentication, API-key handling, token caching, and secure-store plumbing lives in the `edge-common-spring` dependency, not here.

## Build & test

Maven wrapper is not committed; use a local `mvn`.

```bash
mvn clean install          # full build + all tests (unit + IT)
mvn test                   # run tests only
mvn test -Dtest=InventoryServiceTest              # single unit test class
mvn test -Dtest=InventoryControllerIT#getInstance_positive   # single test method
mvn spring-boot:run        # run locally (requires FOLIO_CLIENT_OKAPIURL etc., see ENV_VARS.md)
```

Test naming (surefire runs all three): `*Test` / `*Tests` are unit tests; `*IT` are integration tests that boot the full Spring context, use MockMvc, and stub downstream FOLIO calls with WireMock (see `BaseIntegrationTests`).

## Architecture

Request flow: **Controller → Service → Feign-style HTTP client → downstream FOLIO service**. Responses are passed through as raw JSON strings (`JsonNode.toString()`) — the module deliberately does *not* deserialize into DTOs for pass-through endpoints, so downstream schema changes don't break it.

- **`controller/InventoryController`** implements `InventoryApi`, the interface generated by the OpenAPI generator from `src/main/resources/swagger.api/edge-inventory.yaml` at build time (into `target/generated-sources`, packages `org.folio.inventory.domain.dto` and `org.folio.inventory.rest.resource`). To add or change an endpoint, edit the YAML first, then implement the generated interface method.
- **`service/`** — `InventoryService` is the thin default path (one call to `InventoryClient`). The `Ecs*` services implement Enhanced Consortia Support: when the request's tenant is the consortium's **central tenant**, holdings/items/inventory-view/locations/material-types are fanned out across member tenants in parallel (`parallelStream` + `FolioExecutionContext.execute` to carry tenant context) and merged. The central-vs-member decision is `EcsInventoryService.isCentralTenant`, checked inside the controller for the affected endpoints only.
- **`client/`** — declarative HTTP interfaces (`@HttpExchange` / `@GetExchange`) for `InventoryClient`, `SearchClient`, `UserClient`, `ConsortiaClient`. They are turned into beans in `config/HttpClientConfiguration` via the `edgeHttpServiceProxyFactory` provided by edge-common-spring. Overloads that take an `@RequestHeader(XOkapiHeaders.TENANT)` are how ECS fan-out targets a specific member tenant. `DataExportService` uses a plain `RestClient` (`dataExportRestClient`) instead, because it streams binary MARC downloads.
- **`handler/InventoryErrorHandler`** — `@RestControllerAdvice` that maps downstream `HttpStatusCodeException` (and validation errors) into the generated `Error` DTO, preserving the downstream status and body.

## Conventions & gotchas

- JSON uses **Jackson 3** (`tools.jackson.databind.*`), not the classic `com.fasterxml.jackson.*` — match imports to existing files.
- Lombok is used throughout (`@RequiredArgsConstructor`, `@Log4j2`); constructor injection only.
- ECS query parsing in `EcsInventoryService` relies on regexes against CQL query strings (instance/holdings id extraction) — changes to accepted query shapes must update those patterns.
- Logging is log4j2 (`log4j2.xml`); logback and `log4j-to-slf4j` are explicitly excluded in `pom.xml`.
- `DataSourceAutoConfiguration` is excluded — this module has no database despite pulling in folio-spring-base.

## FOLIO packaging

`descriptors/ModuleDescriptor-template.json` and `DeploymentDescriptor-template.json` are filtered at `generate-resources` into `target/*.json`; these declare the module's routes, permissions, and env for the FOLIO platform. Institutional users need the `inventory.instances.item.get` permission. Operator-facing environment variables are documented in `ENV_VARS.md`; TLS setup (inbound HTTPS and outbound Feign truststore) is in `README.md`.
